# 记忆治理与人类接口设计

## 1. 文档目标

本文档用于定义 `local-project-memory` 中“人类可维护知识层”和“机器工作记忆层”的边界。

目标不是让开发者直接维护底层 memory 数据，而是明确：

- 哪些知识应由人类直接维护
- 哪些知识应由系统自动生成和消费
- 哪些内容适合进入长期记忆
- 哪些内容只适合停留在临时工作层
- 代码、文档、配置、任务总结四类索引对象在系统中的角色

## 2. 核心判断

系统成熟后，人类开发者真正长期直接接触的，通常是：

- 项目文档
- 项目规则
- 架构说明
- agent 身份与工作规程
- 少量被提炼后的稳定知识

而底层 memory 数据天然更偏机器友好，不偏人类友好。

原因在于，memory 层主要服务：

- recall
- 过滤
- 排序
- 去重
- 合并
- 跨任务复用

因此它会依赖大量中间结构，例如：

- chunk
- metadata
- scope
- confidence
- verified
- citation
- related ids
- access statistics

这些结构对机器很有价值，但不适合作为人类日常维护界面。

## 3. 三层模型

建议把整个系统分成三层，而不是把所有东西都称为“记忆”。

### 3.1 Source of Truth 层

这一层是人类权威来源。

主要包括：

- 正式文档
- 项目规则
- 代码与配置原文
- 团队约定
- agent role spec

特点：

- 可读
- 可评审
- 可讨论
- 可版本管理

### 3.2 Memory / Index 层

这一层是机器工作层。

主要包括：

- `doc_chunk`
- `code_chunk`
- `config_chunk` 或兼容形式的配置记录
- `task_summary`
- metadata
- recall cache
- ranking features
- verified / unverified memory

特点：

- 主要给系统消费
- 主要用于 recall 与长期复用
- 不适合作为团队主要编辑界面

### 3.3 Human Review 层

这一层是人和 memory 之间的治理接口。

主要包括：

- 最近新增的长期记忆
- 待确认的任务结论
- 高频命中的稳定知识
- 可晋升为正式文档的结论
- 疑似过期或冲突的记忆

特点：

- 不直接暴露底层 chunk 细节给人维护
- 让人类可以审核、确认、修正、提升知识

## 4. 人类直接维护什么

建议长期由人类直接维护的内容包括：

- 项目说明文档
- 模块说明文档
- 项目规则与 SOP
- agent 身份定义
- agent 工作规程
- 确认后的稳定架构知识

这些内容应该尽量保持为文档化表达，而不是直接写入底层 memory 表。

## 5. 系统自动维护什么

建议由系统自动生成和维护的内容包括：

- chunk 切分结果
- embedding
- metadata
- recall 特征
- query 命中统计
- 任务中间发现
- 临时工作记忆
- 索引更新状态

这些内容对系统重要，但不应要求开发者手工维护。

## 6. 为什么底层 memory 不应由人直接维护

如果让人类直接维护 memory 表，通常会出现这些问题：

- 表达不自然
- 修改成本高
- 团队难形成一致约定
- 大量字段只对机器有意义
- 维护负担远高于收益

典型不适合人工直接维护的字段包括：

- `id`
- `scope`
- `importance`
- `confidence`
- `related_ids`
- `access_count`
- `line_hint`

因此系统不应把“人工编辑 memory 记录”作为主要治理方式。

## 7. 四类索引对象的定位

第一阶段最关键的四类对象是：

- 代码
- 文档
- 配置
- 任务总结

### 7.1 代码索引

代码索引是开发任务中的主要定位入口。

主要作用：

- 快速定位模块
- 找类、方法、入口点
- 发现功能实现分布
- 把 agent 引导到真实源码验证

人类维护方式：

- 人类维护代码本身
- 系统自动完成代码切块和索引

### 7.2 文档索引

文档索引是稳定背景知识层。

主要作用：

- 提供架构背景
- 提供项目规范
- 提供设计意图
- 连接代码与概念

人类维护方式：

- 人类维护正式文档
- 系统自动切块并建立 recall 入口

### 7.3 配置索引

配置索引是“代码之外的重要行为入口”。

主要作用：

- 暴露依赖和开关
- 暴露包管理与项目设置
- 为查找功能行为提供补充线索

人类维护方式：

- 人类维护配置原文
- 系统自动切块、索引和引用追踪

### 7.4 任务总结索引

任务总结是长期记忆闭环的核心对象。

主要作用：

- 沉淀一次任务的已验证结论
- 复用历史问题入口
- 记录问题根因和有效路径
- 为后续相似任务提供先验

人类维护方式：

- 人类不直接维护底层记录
- agent 或系统在任务结束后生成候选总结
- 人类或规则系统决定其是否进入长期层

## 8. 记忆写入治理

### 8.1 允许直接进入长期层的内容

- 已验证的任务结论
- 用户明确确认的项目规则
- 已确认的 bug 根因
- 稳定入口映射

### 8.2 只允许停留在 session 层的内容

- 临时定位结果
- 尚未验证的怀疑点
- 中间分析草稿
- 一次性调试结论

### 8.3 不应写入长期层的内容

- 全量对话
- 原始思考过程
- 大段重复源码
- 噪声日志
- 没有 citation 的模糊结论

## 9. 人类如何治理 memory

推荐的人类治理动作不是“编辑记录字段”，而是：

- 确认某条结论是否成立
- 将某条结论提升为正式文档
- 标记某条记忆过期
- 合并重复经验
- 修正文档、规则或 agent spec

也就是说，人类更适合维护知识表达层，而不是维护 memory 数据层。

## 10. Agent 如何使用这三层

后续的 agent，尤其是“架构师型 agent”，应按以下原则工作：

1. 先从 Memory / Index 层 recall 候选上下文
2. 再回到 Source of Truth 层读取原文验证
3. 基于验证结果给出实现建议或架构建议
4. 任务结束后只写入经过提炼的总结
5. 稳定结论在适当时机晋升到文档或规则层

这样才能让 agent 表现出长期理解力，而不是伪装成“永远记得一切”。

## 11. 对未来架构师 agent 的意义

如果长期使用这套系统，并配合：

- 完整索引
- 良好写入治理
- 明确的 role spec
- 稳定的工作规程

那么 agent 的表现会越来越像一个长期参与项目的架构师助手。

但要明确：

- 它不是模型内部天然拥有长期记忆
- 它是通过外部 memory/index 层表现出连续理解能力

这个差异决定了系统应优先建设知识治理，而不是只追求“回答像不像懂项目”。

## 12. 落地建议

在当前阶段，建议按以下顺序落地：

1. 继续完善代码、文档、配置、任务总结四类索引对象
2. 明确任务总结的写入准则
3. 增加稳定知识晋升机制
4. 增加人类可审查的 memory 视图
5. 再设计架构师 agent 的 role spec 和 skill contract

## 13. 设计结论

最终系统不应让人类“维护 memory 表”，而应让人类维护：

- 文档
- 规则
- 角色规范
- 稳定知识

系统再把这些内容映射为可检索、可过滤、可复用的 machine-friendly memory 结构。

这才是适合长期演进的设计边界。
