# 项目记忆工作流设计

## 1. 文档目标

本文档定义 `local-project-memory` 的标准工作流，说明系统在一个真实项目中如何完成：

- 项目初始化
- 日常开发中的 recall 使用
- 任务结束后的知识沉淀
- 长期治理与持续增效

本文档不是 API 设计文档，而是面向后续实现、agent skill、验收流程和团队协作的总工作流说明。

## 2. 核心结论

系统成熟后，推荐工作流不是“把所有内容都塞进 memory”，而是一个四步闭环：

1. 项目初始化
2. 任务执行
3. 任务收尾
4. 持续治理

只有四步都成立，memory 的增长才会稳定转化为效率增长。

## 3. 工作流总览

### 3.1 项目初始化

目标：

- 为指定项目建立独立的长期记忆与索引上下文
- 形成该项目的 doc/code/config 初始索引
- 建立 `project_id`、scope 和规则上下文

主要动作：

- 指定 `project_root`
- 确认 docs root 与 code root
- 执行全量索引
- 建立项目级文档、代码、配置索引
- 加载项目规则和 agent spec

输出：

- `project:<project-id>` scope 可用
- 基础 recall 可以命中文档、代码、配置
- 项目进入“可持续记忆”状态

### 3.2 任务执行

目标：

- 在真实开发任务里使用 memory/index 层缩小阅读范围
- 让 agent 优先 recall，再回到源码验证

标准流程：

1. 接收任务
2. 调用 recall 获取候选上下文
3. 读取文档、代码、配置和历史任务总结
4. 缩小范围
5. 回到原文件验证
6. 给出建议、方案或修改

关键原则：

- memory 负责缩小范围，不负责替代源码
- recall 结果必须尽量带 citation
- 高分结果仍然需要原文验证

### 3.3 任务收尾

目标：

- 从已完成任务中提炼可复用知识
- 同步写入 memory 和适合人类维护的知识层

标准流程：

1. 生成候选任务总结
2. 区分 verified / unverified
3. 决定写入 `session`、`project memory` 或正式文档
4. 必要时更新项目规则、SOP、架构说明

关键原则：

- 不是把任务过程原样存档
- 只写入经过提炼且有价值的内容
- memory 和文档是两个目标层，不应混写

### 3.4 持续治理

目标：

- 保持长期记忆可用，而不是只增长体积

主要动作：

- 去重
- 合并相似经验
- 提升稳定知识
- 清理过时内容
- 识别高频命中的有效路径
- 将稳定结论晋升到正式文档或规则层

关键原则：

- memory 的长期价值来自质量治理，不来自条目数量

## 4. 四类索引对象如何进入工作流

当前阶段最关键的四类对象是：

- 代码
- 文档
- 配置
- 任务总结

### 4.1 代码

作用：

- 定位真实实现入口
- 为修改与验证提供最终依据

在工作流中的位置：

- 初始化时建立代码索引
- 执行阶段作为主要 recall 命中对象
- 收尾阶段只写回“关于代码的结论”，不直接把代码复制进 memory

### 4.2 文档

作用：

- 提供背景、规则、设计意图和模块关系

在工作流中的位置：

- 初始化时建立文档索引
- 执行阶段帮助理解背景和约束
- 收尾阶段稳定结论可反向补全文档

### 4.3 配置

作用：

- 提供行为开关、依赖、包管理和项目设置线索

在工作流中的位置：

- 初始化时纳入索引
- 执行阶段辅助定位行为来源
- 收尾阶段只有“配置影响结论”应写回任务总结或规则层

### 4.4 任务总结

作用：

- 形成长期记忆闭环
- 让后续相似任务不必从零开始

在工作流中的位置：

- 执行阶段可作为 recall 对象
- 收尾阶段由当前任务生成
- 治理阶段决定是否晋升为稳定知识

## 5. 推荐的日常使用方式

### 5.1 每个项目独立建库

建议每个项目使用独立的：

- `project_id`
- 索引数据
- 长期记忆上下文
- agent 规则组合

不建议多个项目共用一个无边界大库。

### 5.2 开发时绑定当前项目

任务运行时，agent 应显式使用当前项目的 `project_id` 和默认 scope。

推荐默认 scope：

- `project:<project-id>`

任务中可附加：

- `session:<task-id>`

### 5.3 完成任务后进行知识提炼

任务结束后，应明确区分：

- 哪些内容写入 `session`
- 哪些内容进入 `project memory`
- 哪些内容应回写文档
- 哪些内容应丢弃

## 6. 推荐写回策略

### 6.1 适合写入 memory 的内容

- 已验证的任务结论
- 已确认的问题根因
- 稳定入口映射
- 高价值检索路径
- 经用户确认的项目约定

### 6.2 适合回写正式文档的内容

- 稳定规则
- 模块职责
- SOP
- 经多次验证的架构知识
- 常见问题与解决方式

### 6.3 不适合写入长期层的内容

- 原始对话
- 原始思考过程
- 未验证猜测
- 重复源码
- 工具输出噪声

## 7. 为什么这个工作流会持续增效

如果系统只是“持续写入”，不一定持续增效。

真正让它持续增效的是：

- recall 能逐步命中更多有效上下文
- 重复问题可以复用历史结论
- 稳定知识逐步从任务经验晋升为文档和规则
- agent 身份和工作规程逐步适配项目

因此，长期增效来自：

- 项目索引增长
- 已验证知识增长
- 人类知识层同步完善
- 规则和治理持续改进

## 8. 未来面向架构师 agent 的意义

如果长期使用该工作流，并且：

- 建库完整
- recall 稳定
- 写回有治理
- 角色规范明确

那么 agent 会逐渐表现出这些特征：

- 更快理解项目结构
- 更快定位模块入口
- 更贴近项目现实地给出建议
- 对重复问题有更强的历史连续性
- 更像一个长期参与项目的架构师助手

但这里要明确：

- agent 不是“内部天然记住了项目”
- agent 是通过外部 memory/index 层获得持续项目理解能力

## 10. 当前阶段的落地重点

基于本文档，当前阶段最应该推进的是：

1. 增加稳定可调用的项目索引入口
2. 接入真实 LanceDB 或抽象存储适配层
3. 为真实项目准备第一轮验收题库
4. 明确任务收尾时的写回模板
5. 设计架构师 agent 的 role spec 和 skill contract

## 11. 设计结论

项目的长期工作流不应是：

- 建一个库
- 不停把东西塞进去

而应是：

- 建立项目级索引和记忆上下文
- 在任务中用 recall 缩小范围
- 在任务后提炼真正有价值的知识
- 用治理机制让 memory 持续成长并持续增效

这才是 `local-project-memory` 应支持的标准闭环。
