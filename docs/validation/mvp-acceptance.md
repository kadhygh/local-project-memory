# MVP 验收文档

## 1. 文档目标

本文档定义 `MemoryService` MVP 的验收方式，确保第一版原型验证的是开发效率提升，而不是表面上的“能回答一些问题”。

MVP 验收关注四件事：

- 能否为本地 Unity 项目建立可检索的项目知识层
- 能否在真实开发任务里缩小阅读范围
- 能否把已验证结论写回并在后续任务中复用
- 能否始终带引用并回到源码验证

## 2. 被验收系统

本次验收对象包括：

- `MemoryService`
- `Indexer Pipeline`
- `LanceDB` 数据层
- 面向 agent 的最小 API

本次不要求：

- Unity Runtime 集成
- Unity 编辑器插件
- 实时监听所有资源变化
- 大规模多模态检索

## 3. 验收环境

建议使用一份本地 Unity 项目作为数据源，项目最好满足这些条件：

- 有中等以上规模的 `Assets/**/*.cs`
- 有基础文档或 README
- 有一些配置文件与 `ProjectSettings`
- 至少包含 UI、状态流转、场景切换、配置读取中的两类以上逻辑

第一阶段索引范围建议限定为：

- `Assets/**/*.cs`
- `Packages/manifest.json`
- `ProjectSettings/**/*`
- 项目内 `*.md`
- 可选的 `*.json`、`*.yaml` 配置

第一阶段不要求深度解析：

- `*.unity`
- `*.prefab`
- 图片、音频、模型等资源

## 4. 验收前置条件

验收开始前，系统至少应具备：

- 全量索引入口
- 增量索引入口
- `search.recall`
- `index.upsert`
- `memory.store`
- 基础结构化结果返回

每条可召回记录至少应带：

- `id`
- `type`
- `scope`
- `source_path`
- `title`
- `summary`
- `citation`

## 5. 验收分层

### 5.1 第一层：索引构建验收

目标是确认系统已经形成真实数据层，而不是聊天外壳。

必须验证：

- 能扫描目标 Unity 项目并完成首轮索引
- 能统计已写入的 `doc_chunk`、`code_chunk`、`task_summary`
- 每条记录都具备可追溯来源
- 支持按 `project` scope 检索
- 支持按 `type` 过滤

通过标准：

- 索引过程无阻塞性错误
- 至少能稳定写入文档和代码两类对象
- 召回结果不存在虚构路径

### 5.2 第二层：检索质量验收

目标是验证 recall 对开发定位是否有效。

验收方式：

- 先准备一组固定题库
- 每个问题预先标注“期望命中的模块或文件”
- 单独执行 recall，不进行代码修改

题库建议 15 到 30 题，并混合两类问题：

- 语义问题
- 关键词问题

示例问题：

- “怪物生成逻辑在哪”
- “哪个系统控制角色状态切换”
- “UI 面板统一打开入口在哪”
- “存档初始化的入口在哪”
- “`QuestState` 被哪些模块推进”
- “`InventoryPanel` 是谁实例化的”

建议指标：

- `Top-5 hit rate >= 80%`
- `Top-3` 至少有 1 条真正可用结果
- 返回结果必须带 `source_path`
- 如果存在 `line_hint`，其位置应接近真实入口

### 5.3 第三层：真实开发任务验收

目标是验证系统能否服务真实开发，而不是只做问答。

建议选择 3 到 5 个中小任务，覆盖：

- 新增小功能
- 修复已知 bug
- 小范围重构
- 配置接线调整

任务示例：

- 给某个 UI 面板增加快捷关闭逻辑
- 修复切场景后玩家状态未重置的问题
- 把怪物生成参数从硬编码改为配置驱动
- 给任务系统新增一个失败状态并接入 UI

每个任务都要求 agent 按同一流程执行：

1. 先调用 `search.recall`
2. 再读取少量原文件验证
3. 输出修改方案或直接改代码
4. 任务结束后调用 `memory.store`

建议记录指标：

- 首次命中正确模块所需时间
- 读取文件总数
- 全局搜索次数
- 错误入口次数
- 完成任务总耗时

### 5.4 第四层：长期记忆验收

目标是验证系统是否开始具备跨任务复用能力。

验收方式：

- 在第一批任务结束后写入任务总结
- 再选择 1 到 2 个相关后续任务
- 观察 recall 是否能把历史结论召回到前几条结果

建议写入的内容：

- 已验证的稳定入口
- 已确认的 bug 根因
- 模块依赖关系
- 用户明确确认的项目约定

通过标准：

- 后续相似任务能命中上一次任务的 `task_summary`
- 历史记忆能帮助缩小范围
- 未验证内容不会被误提升为稳定知识

## 6. 对照实验

MVP 验收不能只看主观感觉，建议做最小对照实验。

同一个任务至少做两轮：

- 一轮不接 `MemoryService`
- 一轮接入 `MemoryService`

对比以下指标：

- 首次定位时间
- 读取文件数
- 是否更快命中正确模块
- 是否减少盲搜

如果条件允许，应由同一名操作者在相近时间内执行，减少人为偏差。

## 7. 验收结论标准

MVP 可判定为通过，至少应满足：

- 能索引本地 Unity 项目的文档、代码、配置
- `search.recall` 返回结构化结果且带引用
- 检索题库的 `Top-5 hit rate` 达到可接受水平
- 在真实任务中降低首次定位时间或读取文件数
- 任务总结可被后续相关任务复用
- agent 仍然会回到源码原文验证，而不是直接相信记忆

如果只能做到“像是理解项目”，但不能稳定减少定位成本，则该 MVP 不通过。

## 8. 推荐验收顺序

推荐按以下顺序执行：

1. 对 Unity 项目做一次全量索引
2. 检查记录数、类型分布和引用字段
3. 跑固定题库，评估 recall 质量
4. 执行 3 到 5 个真实开发任务
5. 写入任务总结
6. 执行 1 到 2 个相关后续任务
7. 形成验收报告

## 9. 不合格信号

出现以下情况时，应视为 MVP 尚未达标：

- recall 结果经常没有真实可追溯来源
- Top 结果长期充满噪声
- 仍然需要大范围手工 `grep`
- 任务结束后写入的记忆无法复用
- 历史猜测污染后续任务
- 系统只能做检索展示，无法帮助缩小阅读范围

## 10. 验收产出

每次验收建议保留这些产物：

- 题库与标准答案
- 任务清单
- 关键指标记录
- 失败样例
- 通过与未通过项
- 下一轮修正建议

这样后续每推进一版，都能基于同一套标准回归验证。
