# Unity 开发中使用长期记忆与索引服务的核心需求

## 1. 文档目的

本文档用于定义 Unity 开发期引入长期记忆和索引服务时，真正需要解决的问题。目标不是“搭一个很先进的 AI 服务”，而是明确什么能力能实际提升开发效率、降低盲搜成本，并支撑后续多轮任务与多 agent 协作。

## 2. 目标场景

我们的目标场景不是游戏内 NPC 记忆，而是开发期 LLM 工作流。

主要使用者包括：

- 外部 coding agent
- IDE 或命令行中的 LLM 助手
- 项目级任务编排器
- 后续可能接入的 Unity 编辑器工具

主要任务包括：

- 理解项目结构
- 快速定位相关代码和资源
- 读取已有文档和历史经验
- 生成修改方案
- 在小范围内验证源码和配置
- 记录任务结论并在后续任务中复用

## 3. 为什么需要长期记忆与索引服务

如果没有长期记忆与索引服务，LLM 在 Unity 项目里的典型工作方式是：

- 从用户需求开始猜测入口
- 大范围 `grep`
- 逐个读文件
- 在多轮任务之间丢失大量上下文
- 重复做相同的定位和理解工作

这会导致几个长期问题：

- 首次定位相关模块耗时较高
- 项目越大，检索噪声越高
- 文档、源码、配置、资源之间关系难以统一理解
- 历史任务经验无法被系统化复用
- 多 agent 协作时会重复劳动

因此，我们真正需要的不是“一个能聊天的系统”，而是一个能为 LLM 提供高质量候选上下文的项目知识层。

## 4. 核心需求列表

### 4.1 统一索引多种项目内容

系统需要同时索引以下内容：

- 项目文档
- 代码文件
- 配置文件
- Scene 信息
- Prefab 信息
- ScriptableObject 或数据表摘要
- 历史任务摘要
- 经验证的问题结论

如果系统只能索引代码，而不能同时纳入 doc、scene、prefab、配置与任务经验，就无法形成真正有用的项目知识网络。

### 4.2 支持语义检索与关键词检索并存

Unity 工程中的实际检索既包含语义问题，也包含硬关键词。

典型语义查询：

- “和刷怪有关的逻辑在哪”
- “哪个系统在控制角色状态切换”
- “UI 打开关闭的统一入口在哪”

典型关键词查询：

- 某个类名
- 某个 prefab 名
- 某个 scene 名
- 某个字段名
- 某个任务 ID

因此系统必须支持：

- vector search
- full-text search
- hybrid retrieval

### 4.3 支持结构化 metadata 过滤

仅靠 embedding 不足以支撑工程检索。系统需要支持结构化过滤。

典型过滤维度包括：

- 按项目过滤
- 按模块过滤
- 按 scene 过滤
- 按资源类型过滤
- 按文档类型过滤
- 按时间范围过滤
- 按是否已验证过滤
- 按 agent scope 过滤

### 4.4 支持可追溯引用

LLM 返回结果时，不能只给出自然语言摘要，还必须尽可能附带来源。

最少应支持这些引用信息：

- 文件路径
- 资源路径
- GUID 或稳定标识
- 类名或符号名
- scene / prefab 名
- 文档标题
- chunk ID
- 更新时间

否则 recall 虽然看起来聪明，但无法稳定服务开发任务。

### 4.5 支持长期记忆而不是一次性检索

系统需要具备跨任务保留上下文的能力。

需要保留的长期信息包括：

- 历史任务的结论摘要
- 某类问题的常见入口
- 经验证的 bug 根因
- 模块之间的稳定关系
- 开发约定和团队偏好
- 高频检索路径

这部分内容与静态文档不同，它是 agent 在实际工作中积累出来的经验层。

### 4.6 支持记忆分层

不同内容不应混成同一类数据。

至少需要区分：

- `Documentation`
- `Project Index`
- `Working Memory`
- `Verified Knowledge`

不同层的数据应有不同的写入策略、检索优先级和淘汰规则。

### 4.7 支持 scope 管理

长期记忆服务不能是一个无边界大库。必须支持 scope。

建议的 scope 最少包括：

- `global`
- `project:<id>`
- `module:<name>`
- `scene:<name>`
- `asset:<id>`
- `agent:<role>`
- `session:<task-id>`

scope 的作用包括：

- 降低噪声
- 控制 recall 范围
- 隔离不同任务与不同项目
- 支撑多 agent 协作

### 4.8 支持增量更新

Unity 项目是持续变化的，索引系统不能只支持全量构建。

必须支持至少两种更新方式：

- 手动全量重建
- 基于文件变化或 Git 变化的增量更新

后续如有必要，再增加编辑器事件触发和定时 consolidation。

### 4.9 支持记忆写入治理

系统不能把所有对话、所有日志、所有工具输出都无差别写入长期记忆。

真正需要写入长期记忆的通常是：

- 已完成任务的摘要
- 已验证的问题原因
- 模块关系结论
- 用户明确确认的项目约定
- 成功的检索路径

不应该直接写入长期记忆的内容包括：

- 未验证猜测
- 原始大段对话
- 重复源码
- 临时工具噪声

### 4.10 支持与源码验证工作流结合

长期记忆和索引服务的目标不是替代源码，而是缩小阅读范围。

理想工作流应为：

- 先 recall 候选上下文
- 再选定少量相关文件
- 最后回到源码、配置、文档原文验证

因此系统必须天然服务“定位”与“筛选”，而不是伪装成最终真相来源。

## 5. 数据对象需求

建议第一阶段支持以下对象类型：

- `doc_chunk`
- `code_chunk`
- `scene_summary`
- `prefab_summary`
- `config_chunk`
- `task_summary`
- `issue_root_cause`
- `project_rule`
- `entity_relation`

这些对象足以覆盖开发期 LLM 的大部分高价值信息。

## 6. 检索结果质量需求

系统返回结果时，应优先满足以下要求：

- 相关性高于覆盖面
- 结果可解释
- 结果可追踪
- 结果尽量去重
- 尽量混合不同来源而不是单一类型淹没结果集

如果一个查询返回 20 个高度相似的代码块，而没有相关 doc、scene、prefab 摘要或历史任务结论，用户体验通常并不好。

## 7. 记忆写入需求

建议对写入做等级划分。

### 7.1 应立即写入的内容

- 任务完成总结
- 用户确认的关键规则
- 已验证的 bug 根因
- 模块和入口的稳定映射

### 7.2 需经过 consolidation 的内容

- 重复出现的临时发现
- 多次任务中反复命中的检索路径
- 多条相似经验的合并摘要

### 7.3 不写入长期层的内容

- 原始思考过程
- 每轮完整对话
- 未落地的尝试性推测
- 无引用的模糊结论

## 8. 评估指标需求

系统上线后，至少需要用以下维度评估价值：

- 平均一次任务首次命中正确模块所需时间
- 平均一次任务读取的文件数
- recall 命中的前 5 条是否有效
- 重复问题是否能快速命中历史结论
- 多轮任务中上下文连续性是否明显提升
- agent 对全局 `grep` 的依赖是否下降

如果这些指标没有改善，说明系统在“看起来聪明”，但没有真正提高开发效率。

## 9. 第一阶段不纳入范围的内容

为了保证推进效率，第一阶段建议暂不纳入以下内容：

- 游戏内 NPC 长期记忆
- Unity runtime 直接深度集成
- 全自动实时索引所有编辑器变更
- 大规模多模态图片检索
- 高自治 agent 行为控制

第一阶段的重点应是开发期 LLM 的知识召回与长期记忆，而不是产品化一切能力。

## 10. 推荐的目标定义

第一阶段可以把目标定义为：

构建一个面向 Unity 开发期 LLM 的项目知识与长期记忆服务，使其能够在代码、文档、配置、scene、prefab 和历史任务之间快速建立关联，优先通过 recall 缩小范围，再通过精确读取完成验证和修改。

## 11. 本文档对应的后续动作

基于本文档，后续应优先推进：

- 明确 schema
- 明确对象类型与 metadata
- 设计索引流水线
- 设计 recall API
- 设计 memory write policy
- 设计与 LLM agent 的交互协议
